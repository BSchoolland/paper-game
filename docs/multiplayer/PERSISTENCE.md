# Durable Persistence + Run-Resume — Design (supersedes ruling R13)

> **Product decision:** run/inventory/overworld state is **durable across server restarts**
> (overrides R13 in-memory-only). **Boundary:** only the durable overworld/run/inventory
> layer is persisted; the in-combat `GameState`/`EncounterSession`/`DefendRound`/`MovementVote`
> stay transient. A mid-combat server death resumes the party **at the overworld departure tile**;
> the encounter is simply re-entered.

## v1 implementation calls on the 4 open items (decided, not product-level)
1. **Server secret** — single `GAME_TOKEN_SECRET` env (>=256-bit). Fail-fast in prod if unset; fixed dev fallback otherwise. **No secret rotation in v1** (rotation grace window + version tag deferred to a follow-up).
2. **Seat lookup key** — use **raw `client_id`** for the `run_seats` lookup + unique-live index (the design's documented safe default). The `client_lookup` HMAC at-rest hardening (R33) is **deferred**; for v1 drop the `client_lookup` column and index/look-up on `client_id` directly. `token_salt` is **kept** (it salts the session-token HMAC, which is independent of lookup).
3. **Housekeeping** — in-process `setInterval` keyed on `updated_at`, retention **N = 7 days**; deactivate/erase stale runs. Lowest-friction v1.
4. **Hex icons** — persist only non-derivable icons in `explored_hex_icons`; recompute derivable structure icons via `getHexIcon` at reconstruction. Phase 3/6 must verify every icon source is persisted OR deterministically derivable.


## Revised rulings (R13 rev + R29-R36)

## R13 (REWRITTEN) — DISCOVERY is GLOBAL-per-dimension; CLEARED-THIS-RUN is per-run; the overworld/run/inventory layer is durable across restart

> **CORRECTION (schema v3).** Earlier revisions of this doc run-scoped exploration (`explored_hexes(run_id,...)`). That was WRONG: it conflated two distinct concepts. **DISCOVERY** (the community fog-of-war: which hexes *anyone, ever* revealed) is **GLOBAL per dimension**, permanent, append-only, shared by ALL runs/rooms — discovering a never-before-seen hex is a key moment. **CLEARED-THIS-RUN** (the party's combat progress this expedition: the durable `visitedThisRun`) is **PER-RUN** and is the ONLY combat-vs-free-move gate. v3 splits them into `discovered_hexes(dimension_id,q,r)` / `discovered_hex_icons(dimension_id,q,r)` (global) and `run_cleared_hexes(run_id,q,r)` (per-run). The text below is updated to this model; the run/inventory durability is unchanged.

The run/inventory layer is `runId`-scoped (migration via `PRAGMA user_version`). The community discovery map is `dimensionId`-scoped. Two concurrent rooms in the same dimension SHARE the community map but each re-fights its own path. We accept a one-time wipe of the old run-scoped exploration tables (documented in PR).

**This ruling now makes the durable overworld/run/inventory layer survive a server restart** (overrides the prior "in-memory only / fresh run / known follow-up" clause). The unit of durability is the **run**, not the connection and not the room code. A run is a long-lived play session for a fixed party on a fixed dimension. The room code (R20, 6-char, quarantined on reap) stays purely in-memory and is **never** the resume key — codes are ephemeral invites. The resume keys are `clientId` (per-seat, from localStorage) and the durable `runs.id` it is bound to.

### R13.1 — Durable vs transient boundary (the explicit line)

DURABLE (SQLite, survives restart): the `runs` row (incl. `party_q/party_r`, `host_client_id`, `active`, `capacity`, `dimension_id`, `outcome`); `discovered_hexes(dimension_id,q,r)` + `discovered_hex_icons(dimension_id,q,r,icon)` (the GLOBAL community fog-of-war + explicitly-set icons, shared across runs/rooms in the dimension); `run_cleared_hexes(run_id,q,r)` (the per-hex **cleared/combat-resolved** set = durable `visitedThisRun`); `run_seats(run_id,seat_index,client_id,display_name,controller_kind,token_salt,joined_at,left_at)`; `run_seat_items(run_id,seat_index,location,slot_order,item_id)`; `run_seat_attachments(run_id,seat_index,item_id,attachment_json)`. Right-to-erasure (R33) deletes the per-run rows but NEVER the global community map (shared, non-personal world state).

TRANSIENT (in-memory only, wiped by restart, rebuilt on demand): `EncounterSession`, `GameState`, `heroBrains`, the combat phase machine, `DefendRound`, `MovementVote`, `aiPlayerBusy`/`phaseTransitioning`/`building`/`generation`, all timers (reap/afk/grace/vote/defend), `RoomCode`, `tokenIndex`, `Seat.socket`, `Seat.ready/exhausted/actedThisPhase`. **`room.phase==="combat"` is intentionally NOT durable.** A restart mid-combat reconstructs the room at `overworld` with the party on `runs.party_q/party_r` (the departure tile); the in-progress encounter is discarded and re-proposing the move re-enters it via the R7 atomic build. Hero death/loot inside an encounter are committed only at `combatEnd{won}` (write point 4) / run-swap (write point 9); a mid-combat crash discards them, by design (surface a client "encounter restarted" banner on resume-into-overworld).

### R13.2 — DISCOVERY (global visibility) and CLEARED-THIS-RUN (per-run combat-resolved) are DISTINCT sets, keyed differently (resolves Finding "visitedThisRun not persisted", HIGH)

The live combat-vs-free-move gate is `visitedThisRun.has(tk)` — **per-run**, NOT discovery: `freshVisitedSet()` seeds only the origin, while `seedDiscovery(dimensionId, 15)` seeds a radius-15 disc into the **global community** map as *visible/discovered*. Conflating them breaks the game loop (the whole pre-seeded disc would become combat-free) or re-triggers combat on already-cleared hexes. Crucially, discovery is shared across runs: a hex the community already revealed is still un-cleared for a fresh run, so that run re-fights it.

Therefore: persist BOTH sets, distinctly and with different keys.
- `discovered_hexes(dimension_id,q,r)` — the GLOBAL, append-only community visibility map. `loadDiscoveredHexes(dimensionId)` drives the hex map every room in that dimension sees. `discoverHex(dimensionId,coord)` returns true iff it inserted a NEW row (first-ever discovery → `hexDiscovered` KEY MOMENT).
- `run_cleared_hexes(run_id,q,r)` — the per-run combat-resolved set. The **only** gate for re-combat. Reconstruction rebuilds `room.visitedThisRun` strictly from `loadRunCleared(runId)` — **never** from visibility.
- Origin is `markRunCleared(runId, ORIGIN)` at run-create (matches `freshVisitedSet()` containing ORIGIN_KEY) and `discoverHex(dimensionId, ORIGIN)` + radius-15 `seedDiscovery(dimensionId, 15)` for the community map. `resetToOrigin` re-clears only the new run's origin; it does NOT re-seed the (persistent) global map.
- `exploreHex` (write point 4) calls `commitExplore(dimensionId, runId, coord, icon)`: one transaction that discovers the hex globally (+ icon), `markRunCleared`s it for THIS run, and advances party position; it returns the first-ever flag. A YES-vote move onto an already-cleared (this-run) hex is a pure party-position move (write point 5), no combat.

### R13.3 — Inventory durability by item ID + ordering (not serialized definitions)

`InventoryState` holds full `ItemDefinition` objects, fully reconstructable from `loadItems(dimensionId)` keyed by `id` (as `buildDefaultInventory` already does). Persist IDs + ordering only:
- `run_seat_items`: `bag` rows use `slot_order` = bag index 0..15 (empty bag slots have no row); `equipped` rows use `slot_order` = equipped-array index (ordering matters for `getAnimSet`/slot accounting).
- `run_seat_attachments`: the `AttachmentData` blob keyed by `item_id` (mirrors in-memory `attachments: Record<itemId, AttachmentData>`).

Rehydration `loadSeatInventory(runId, seatIndex, dimensionId) -> InventoryState`: `const merged = {...loadItems(0), ...loadItems(1), ...loadItems(2), ...loadItems(3), ...loadItems(dimensionId)}` (same merge as today), then build length-16 `bag` (nulls where no row), ordered `equipped`, and `attachments` map. **Item-not-found guard:** if a persisted `item_id` is absent from `merged` (content edit across deploy), skip + log; the seat loses that one item rather than crashing resume. The equipped-array index is recomputed from surviving rows so slot accounting stays valid.

### R13.4 — Attachments are id-scoped by design (resolves Finding "duplicate-item-id attachment collision", MED)

The in-memory model already keys attachments by `item_id` (`Record<string, AttachmentData>`), and `unequipItem` does `delete attachments[removedItem.id]` — i.e. one attachment config per item id per seat, shared across all copies of that id. The persistence layer **faithfully reproduces this** and does not introduce a per-instance attachment. This is the documented contract: **attachments are id-scoped, not instance-scoped.** `run_seat_attachments` PK `(run_id, seat_index, item_id)` matches the in-memory cardinality exactly, so a seat carrying two copies of the same id shares one attachment config (the existing behavior). A bind-time `validateLoadout` MUST assert no attachment row references an `item_id` not present in that seat's `run_seat_items`. A determinism test with two same-id items asserts rehydration is deterministic.

### R13.5 — Run lifecycle / inactivation matrix (`runs.active` is the single liveness flag)

- **Victory ends the run** (`gameOver outcome="victory"`): one transaction sets `active=0, outcome='victory', completed_at=now` AND `left_at`-stamps EVERY `run_seats` row for that run (see R32). Does NOT auto-create a new run (party returns to lobby/menu; "play again" makes a fresh run via createRoom).
- **Party wipe / defeat** (today's `resetToOrigin` new-run path): one transaction — old run `active=0, outcome='defeat'`, `left_at`-stamp all its seats, INSERT new `runs` row (same dimension, same human party copied with fresh starter inventory, party at ORIGIN, `active=1`), `markRunCleared(newRunId, ORIGIN)`, and repoint `room.runId` + re-key the transient `Map<runId,Room>` (R30). The GLOBAL community discovery map is untouched (it persists across runs); the new run rebuilds `room.hexMap.hexes` from `loadDiscoveredHexes(dimensionId)`.
- **Explicit host reset in overworld**: same as defeat (`outcome='abandoned'`, new active run). In-combat `reset` only aborts the encounter and returns to overworld of the SAME run (no run swap, no durable change; matches R17).
- **Abandonment** (all humans leave, no run-end event): the run stays `active=1` so a returning player resumes; a reap timer (R31) eventually disposes the room object. An optional housekeeping job inactivates+deletes runs past N days (R33).

### R13.6 — Write points (every durable write; single-statement writes are atomic-by-being-single-statement; multi-statement/multi-table writes use `db.transaction`)

Each write fires AFTER the in-memory mutation succeeds (durable is a committed subset of in-memory), **except** equip (R34: write-before-ack). bun:sqlite is synchronous; no async gap.

1. **Run create** (`createRoom`): INSERT `runs`(active=1, dimension_id, capacity, host_client_id=creator, party_q/r=ORIGIN, created_at/updated_at=now); `seedDiscovery(dimensionId, 15)` + `discoverHex(dimensionId, ORIGIN)` + `saveDiscoveredHexIcon(dimensionId, ORIGIN, 'town')` into the GLOBAL community map (idempotent — a returning dimension keeps prior discoveries); `markRunCleared(runId, ORIGIN)` for this run; INSERT `run_seats` host row (controller_kind='human', client_id=creator, fresh token_salt) + its starter `run_seat_items`. Before all this, run the R32 prior-run cleanup for the creator's clientId.
2. **Seat bind / join / bot-fill**: one transaction per seat — UPSERT `run_seats`(client_id, display_name, controller_kind, token_salt on first human bind) and (on first human bind) snapshot starter inventory into `run_seat_items`. Bot-fill writes controller_kind='bot', client_id=NULL, token_salt=NULL. Run R32 prior-run cleanup for a joining human's clientId in the same transaction.
3. **Equip / unequip / updateAttachment** (legal only while `room.phase!=="combat"`, R26): one transaction — gate on R29 identity re-check, then DELETE `run_seat_items` for (run_id,seat_index), re-INSERT bag+equipped rows by item id+order, upsert/delete matching `run_seat_attachments`. Commit BEFORE sending the `inventory` ack (R34).
4. **Hex explore** (`exploreHex` on `combatEnd{won:true}` with `pendingHex`): one transaction (`commitExplore(dimensionId, runId, coord, icon)`) — `INSERT OR IGNORE discovered_hexes(dimension_id,q,r)` (+ icon row), `INSERT OR IGNORE run_cleared_hexes(run_id,q,r)`, AND `UPDATE runs SET party_q=?, party_r=?, updated_at=?`. Global discovery, the per-run cleared set, and party position advance atomically; the returned first-ever flag drives the `hexDiscovered` KEY-MOMENT broadcast.
5. **Party move to an already-cleared hex** (vote YES, no combat): single `UPDATE runs SET party_q=?, party_r=?, updated_at=?`. Written **synchronously in the same handler turn as the in-memory move, before yielding to the event loop** (R35), so durable party_q/r never lags in-memory playerPos when the next combat-entry message is processed.
6. (folded into 4/5 above)
7. **Run complete (victory)** / **defeat** / **reset** — see R13.5 matrix; transactional.
8. **Host change** (`migrateHost` or host leaves lobby): single `UPDATE runs SET host_client_id=?`. Convenience write only — host is re-derived from live sockets on resume (R14) regardless.
9. **Client switches to a new room while holding an active run**: see R32 — folded into the createRoom/joinRoom transaction.

NOT write points (transient): combat entry (R7 atomic build), defend, action, vote, ready/pass, host migration's in-memory part, reconnect rebind.

### R13.7 — Migration

Bump `PRAGMA user_version` (now `3`) to a single new version that, in one transaction: ALTERs `runs` to add `dimension_id, capacity, host_client_id, active, party_q, party_r, created_at, updated_at, completed_at, outcome` (guarded try/catch, rethrow non-"duplicate column"); DROPs the old run-scoped `explored_hexes`/`explored_hex_icons`; CREATEs `discovered_hexes(dimension_id,q,r)` + `discovered_hex_icons(dimension_id,q,r)` (GLOBAL) and `run_cleared_hexes(run_id,q,r)` (per-run); creates `run_seats`, `run_seat_items`, `run_seat_attachments` and the R32 indices/constraints. Guard each ALTER in try/catch (matching the existing `dimensions` ALTER pattern). The one-time exploration-table wipe is documented in the PR. Test against `:memory:` (R23).

## R29 (NEW, HIGH) — Per-seat secret salt; durable binding is authoritative; reclaim is token-gated with NO no-token fallback for human seats

Resolves the two CRITICAL identity findings (no-salt enumerable HMAC; no-token admission escape hatch) and the HIGH cross-seat durable-inventory-write finding.

**Token derivation (replaces R5's nonce-only and the delta's no-salt HMAC):**
`sessionToken = HMAC(SERVER_SECRET, clientId + ":" + runId + ":s" + seatIndex + ":" + token_salt)`, where `token_salt` is a random ≥16-byte value persisted in `run_seats.token_salt`, written at first human bind. All inputs are durable, so the token survives restart; the salt keeps each seat **individually unguessable** so a `SERVER_SECRET` leak is not trivially enumerable forgery (clientId is presented by the attacker, runId is a small AUTOINCREMENT int, seatIndex is 0..3). The pure derivable HMAC-without-salt is **removed entirely**. `SERVER_SECRET` (`GAME_TOKEN_SECRET`) MUST be ≥256 bits of entropy (generated, not a passphrase); the server fails fast at boot in production if unset or shorter, and uses a fixed dev secret otherwise. The token travels only in the first `hello`/`reclaim` (R5/R22), never in a URL; TLS is mandated for the handshake in production.

**Reclaim authorization (replaces the delta's resume-step-4 "treat as fresh if no token presented"):**
For any `run_seats` row with `controller_kind='human'`, reclaim **requires a valid token, period** — `if (row.controller_kind==='human' && !validToken) reject`. The no-token admission branch is **deleted**. An owner who genuinely cleared storage and lost the token does NOT silently inherit the seat; the seat stays `human-disconnected`, is bot-driven for liveness (R31), and is recoverable only via host re-invite / explicit forfeit — never first-come admission on a known clientId.

**`clientId` is NOT a secret;** it gates *which* seat may be reclaimed (`findActiveSeatForClient`), the token gates *the right to take it*. `findActiveSeatForClient` resolving to a row is necessary but not sufficient.

**Durable inventory writes are re-authorized at write time:** every equip/unequip/updateAttachment transaction (write point 3) asserts, in the same transaction, `socket.data.clientId === (SELECT client_id FROM run_seats WHERE run_id=? AND seat_index=? AND left_at IS NULL)` and rejects otherwise. A mis-bound socket (force-takeover sequencing, race) can therefore never durably overwrite the legitimate owner's bag. On force-takeover, the displaced socket's `data.seatId` is cleared BEFORE `displaced` is sent so it can no longer issue seat-scoped writes.

**Constant-time, non-enumerating verification (resolves Finding "active-run enumeration oracle", LOW):** verification always performs a constant-time compare, **including a dummy compare when `findActiveSeatForClient` returns null**, and returns an identical-shape welcome-to-lobby payload whether the clientId is unknown, has no active run, or presented a bad token. No observable "this clientId has a run" signal.

**Secret rotation:** a persisted `secret_version` tag lets the server accept old-version tokens for a documented grace window rather than an all-or-nothing flag day. Rotation = (rare, documented) forced re-handshake once the grace window lapses.

## R30 (NEW, HIGH) — Single-Room-per-run guard is the PRIMARY anti-split-brain mechanism; run-swap re-keys it atomically

Resolves the HIGH "two reconnects build two Rooms" risk, the MED "run-swap re-key split-brain", and the LOW "no-await invariant is fragile/convention-only".

The transient `Map<runId, Room>` is the authoritative idempotency registry. **`registerRoomForRun(runId, room)` is a check-or-throw:** it throws (or returns the existing Room) if a *different* Room is already registered for that runId. This is the PRIMARY guard and survives a future async refactor; the "reconstruction contains zero `await`" property (bun:sqlite reads are synchronous) is kept only as defense-in-depth, asserted by a dev check/test that `reconstructRoomForRun` performs no await. Two near-simultaneous reconnects therefore: the first synchronously builds + registers, the second's `register` loses cleanly and it reuses the existing Room; different `seat_index` → no seat contention.

**Run-swap (defeat/reset, R13.5) re-keys atomically:** in the same synchronous block that sets `room.runId = newRunId`, delete the old-runId entry and `registerRoomForRun(newRunId, room)` (moving the existing Room, not orphaning it). No `await` between the DB run-swap transaction commit and the in-memory repoint+re-key. A reconnect that resolves `findActiveSeatForClient` to the NEW run thus always finds the single live Room.

## R31 (NEW, HIGH) — Reconstructed-disconnected seats get liveness from t0; reclaim-with-valid-token always beats a pending bot-flip; reconstructed rooms arm a reap timer

Resolves the HIGH "reconstructed human-disconnected seat deadlocks maybeEndPlayerPhase / R28 grace never armed" and the LOW "host-less reconstructed room lingers / arm reap at reconstruction".

A seat reconstructed as `controller_kind='human'` is set `state='human-disconnected'`, `socket=null`, and — critically — is given a **transient bot brain (`sovereignFor(seat)`) from t0** so `maybeEndPlayerPhase` (R8) can always complete and overworld votes (R15) never wait on a never-present human. The seat is simultaneously **reclaim-only** (R21: never `open`) and **bot-driven for liveness**. This mirrors the post-grace disconnect state (R9/R28) but is established at reconstruction rather than via a `close()` event (which never fired across a restart). State stays `human-disconnected` for roster display; the brain drives combat.

**Reclaim vs flip race is fully defined:** a reclaim presenting a valid token (R29) ALWAYS wins over the bot brain — on reclaim, drop the bot brain (R12), bind the socket, set `state='human-connected'`, apply R10 ready/exhausted rules. A pending flip/bot-drive never invalidates the durable token and never blocks a legitimate reclaim. (The R28 3s grace value is unchanged for *live* disconnects; reconstructed seats skip the grace and are bot-driven immediately, since there was never a live socket to ride a flaky connection back.)

**Reap on reconstruction:** `reconstructRoomForRun` arms the idle reap timer immediately (not only when the last human leaves), so a reconstructed room nobody reconnects to is torn down (room object disposed; the run stays `active=1` and resumable later) rather than lingering forever holding a code. The first human bind both derives host (R14) and cancels the reap timer in the SAME synchronous block.

## R32 (NEW, HIGH) — `run_seats` binding integrity: DB-enforced single-valued live seat, transactional swap, deterministic resolution, consistency CHECK

Resolves the two HIGH binding-integrity findings (clientId in two active runs / two seats one clientId; partial-unique-index crash) and the MED "human row with NULL client_id is unreclaimable + undrivable" and the MED "win-then-createRoom dangling left_at=NULL crash".

**DB-enforced uniqueness (excluding bots):**
`CREATE UNIQUE INDEX idx_run_seats_client_live ON run_seats(client_id) WHERE client_id IS NOT NULL AND left_at IS NULL;`
(The `client_id IS NOT NULL` predicate excludes bot rows so multiple bot seats — all `client_id=NULL` — never collide.)

**Single transaction for the swap:** the prior-seat `left_at`-stamp + (if the prior run thereby has zero live human seats) prior-run `active=0, outcome='abandoned'` + new-seat INSERT all happen in ONE `db.transaction`, with the stamp ordered first, so there is never a window with two live rows for one clientId. This is the single owner of the clientId→active-seat mapping; createRoom/joinRoom for a client with an active run route through this same path.

**`left_at`-stamp ALL seats on every run-end:** victory/defeat/reset (R13.5) `left_at`-stamp every `run_seats` row of the ending run in the same transaction that sets `active=0`. This eliminates the win-then-play-again crash: no `left_at=NULL` row survives against an inactive run, so the unique index can never see two live rows for the client after a victory.

**Deterministic resolution:** `findActiveSeatForClient` is `... WHERE run_seats.client_id=? AND runs.active=1 AND run_seats.left_at IS NULL ORDER BY runs.updated_at DESC, runs.id DESC LIMIT 1` — so even a residual duplicate resolves predictably to the newest run.

**Consistency CHECK (no phantom/dead human seats):**
`CHECK((controller_kind='bot' AND client_id IS NULL) OR (controller_kind='human' AND client_id IS NOT NULL))`
so a `controller_kind='human'` row with `client_id=NULL` (unreclaimable AND, absent R31, undrivable) cannot exist. Reconstruction additionally treats any human seat whose reclaim is provably impossible as bot-driven (R31). Binds/abandons are fully transactional so a partial write cannot leave a half-bound human row.

**Cross-Room identity teardown (resolves MED "same clientId live in two in-memory Rooms"):** when the swap `left_at`-stamps a client's prior active seat, it ALSO looks up any in-memory reconstructed Room for that prior runId via `Map<runId,Room>` and, if that seat has a live socket, closes it / sends `displaced` and unbinds — making the DB abandonment and the in-memory teardown atomic at the application level so R6's single-live-socket invariant holds across the two registries (created-by-code vs reconstructed-by-runId).

## R33 (NEW, MED) — Retention / right-to-erasure (privacy)

Resolves the MED GDPR finding (`clientId`+`display_name`+behavioral profile persisted indefinitely with no deletion path).

`clientId` is a stable pseudonymous device identifier and is personal data. Therefore:
- **Erasure routine** `eraseClient(clientId)`: deletes/anonymizes `run_seats`, `run_seat_items`, `run_seat_attachments`, and `runs` rows for that clientId (right-to-erasure). Provide a server admin entry point.
- **Retention housekeeping** (this PR ships the job, not "optional"): a periodic pass marks runs `active=0` and DELETEs completed/abandoned runs (and their cascade) past N days, keyed on `completed_at`/`updated_at`. The same pass also catches the "abandoned run stays `active=1` forever" case via an `updated_at`-based idle threshold.
- **At-rest lookup hardening:** store `client_lookup = HMAC(SERVER_SECRET, clientId)` as the indexed lookup key in `run_seats` instead of (or alongside) the raw UUID, so a DB dump alone is not a clean device-tracking dataset. Equality lookup (`findActiveSeatForClient`) uses `client_lookup`; this is co-designed with R29 token derivation (the token still binds the raw clientId presented in `hello`, which the server has in memory only for the live session). Document this as the one interaction point between the privacy and identity layers.

## R34 (NEW, LOW) — Equip durability ordering: write-before-ack closes the lost-last-equip window

Resolves the LOW "lost-last-equip-on-crash; reconstructed combat re-fight uses stale loadout".

Today equip mutates in-memory then `sendInventory`. Since bun:sqlite is synchronous and the durable write (write point 3) fires in the same handler turn, **order the DB commit BEFORE the `inventory` ack** so the client UI never reflects an equip the DB has not persisted. The window where in-memory leads durable shrinks to a single synchronous handler turn (effectively zero). Documented explicitly: a crash strictly between the in-memory mutation and the (immediately following, same-turn) commit can lose one in-flight equip; this is acceptable and bounded to lobby cadence.

---

## R35 (NEW, MED) — Party-position write barrier: durable party_q/r never lags in-memory playerPos at combat entry

Resolves the MED "visited-hex move not committed before combat entry mis-places party on resume".

Combat entry (R7) does no durable write and reads the in-memory party position. A YES-vote move onto an already-cleared hex (write point 5) advances in-memory `playerPos`; its single-statement `UPDATE runs SET party_q/r` MUST be issued **synchronously in the same handler turn as the in-memory move, before returning to the event loop** where the next (combat-triggering) message could be processed. Because bun:sqlite writes are synchronous, this guarantees durable `party_q/r` equals in-memory `playerPos` at the instant combat entry flips the phase. A crash during the subsequent encounter build (no durable write) thus leaves durable `party_q/r` on the correct departure tile, so resume + re-propose re-enters from the same adjacency. Stated invariant: **in-memory `playerPos` may never lead durable `party_q/r` across an event-loop yield.**

## R36 (NEW, MED) — Room codes do not survive restart (documented), reconstructed code never aliases a live code

Resolves the MED "stale invite code aliases a stranger's room after restart (quarantine is transient)".

Documented contract: **invite codes do not survive a restart.** A stale pre-crash code is invalid (resolves `ROOM_NOT_FOUND`) until coincidentally recycled from the 32^6 space; resumers reach their run via `clientId`/`hello`, never via code, and the (re-derived) host re-shares the new code. Hard guard: a reconstructed/new Room MUST never be assigned a code currently held by another live in-memory Room (assert at registry insert). Optional (out of scope) hardening: persist quarantine as `(code, expires_at)` rows to suppress cross-restart reuse for the TTL.

## Resume algorithm (canonical, folds R29–R36 into DESIGN §7)

On `hello`/reconnect after a restart (in-memory rooms/tokens gone, durable run exists):
1. `hello{clientId, protocolVersion[, token]}`. Validate protocol; mismatch → `protocolMismatch` + close.
2. Fast path: in-memory `tokenIndex`/registry (empty after restart) → fall through (R19).
3. DB path: `row = findActiveSeatForClient(clientId)` (R32 deterministic). If null → ALWAYS run a dummy constant-time HMAC compare (R29) and return identical-shape welcome-to-lobby (no enumeration signal).
4. Row found → `{runId, seatIndex, dimensionId, capacity, token_salt}`. Compute `expectedToken = HMAC(SERVER_SECRET, clientId+":"+runId+":s"+seatIndex+":"+token_salt)` (R29). For `controller_kind='human'`: a valid token is REQUIRED — no token or mismatched token → reject (no "treat as fresh"), EXCEPT the run-swap reconciliation: if the mismatch is explained by this same clientId's own run rotation (its prior token bound an inactive predecessor run for the same clientId), do NOT reject — re-derive the new token and issue a fresh `welcome` so the client adopts it (the durable binding is authoritative for the client's own seat after a legitimate defeat/reset; this never relaxes the live-room same-run R6 hijack case). Always return the (re-derived) token in `welcome` so client and server converge.
5. Reconstruction (idempotent): `room = Map<runId,Room>.get(runId)`. If absent, `reconstructRoomForRun(runId)` synchronously (no await): read `runs`, `run_seats` (ORDER BY seat_index), `loadDiscoveredHexes(dimensionId)` (GLOBAL visibility) + `loadDiscoveredHexIcons(dimensionId)`, `loadRunCleared(runId)` → `visitedThisRun` (per-run, R13.2), per-seat `loadSeatInventory`. Build a fresh Room: `phase='overworld'`, `session=null`, `vote=null`, `defendRound=null`, `generation=0`, `building=false`, `pendingHex=null`, `hexMap.playerPos={party_q,party_r}`. For each seat: `controller_kind='human'` → `state='human-disconnected'`, `socket=null`, transient `sovereignFor(seat)` brain from t0 (R31); `controller_kind='bot'` → `state='bot'`, bot brain. Assign a NEW RoomCode (R36). `registerRoomForRun(runId, room)` check-or-throw (R30); register by code too. Arm the reap timer (R31). If present, reuse it.
6. Bind socket: `seat.socket=this`, `seat.state='human-connected'`, drop the seat's bot brain (R31), cancel reap timer, re-derive host (R14).
7. Re-derive host (R14 stays-migrated): first human to bind → host (lowest-index connected); persisted `host_client_id` is a seed/tiebreak only if it matches a currently-connected seat. Persist (write point 8).
8. Send `welcome{sessionToken, reconnected:{code,seatId}}`, then `roomState`, `hexMapState` (party on durable tile), `inventory`. Do NOT send combat state/`coopStatus` (phase is overworld).

## Full design

## Durable Persistence + Run-Resume Delta

### 0. Scope and boundary (overrides R13's "in-memory only / fresh run")

We make **the durable overworld/run/inventory layer survive a server restart**, while keeping the transient in-combat `GameState`/`EncounterSession`/`DefendRound`/`MovementVote` strictly in-memory. The persistence layer is the *Room reconstruction substrate*: enough rows to rebuild a `Room` at the **overworld** phase with the correct seats, controllers, host, party tile, explored hexes, and per-seat bags — but never a live combat snapshot.

The unit of durability is the **run**, not the connection and not the room code. A run is a long-lived play session for a fixed party on a fixed dimension. The room code (R20, 6-char, quarantined on reap) stays purely in-memory and is *never* the resume key — codes are ephemeral invites. Resume keys are `clientId` (per-seat) and the durable `runs.id` they are bound to.

This is layered cleanly under the existing rulings: it does not touch the pure resolver (R1/R2), does not add fields to `GameState`, and slots into the Phase-3 run-scoped DB plus the Phase-6/7 identity flow. The only conceptual change to R13 is "per-seat inventory and the run survive a restart" plus a new `clientId<->seat<->run` binding table; everything else in R13's run-scoping stands.

### 1. Durable vs. transient (the explicit line)

DURABLE (SQLite, survives restart):
- The run itself: `runs(id, dimension_id, capacity, host_client_id, active, party_q, party_r, created_at, updated_at, completed_at, outcome)`.
- Explored hexes, run-scoped: `explored_hexes(run_id, q, r)` (already Phase-3) + their icons.
- Per-seat membership/binding: `run_seats(run_id, seat_index, client_id, display_name, controller_kind, joined_at, left_at)`.
- Per-seat inventory, by **item ID** not full definition: `run_seat_items(run_id, seat_index, location, slot_order, item_id)` + `run_seat_attachments(run_id, seat_index, item_id, attachment_json)`.

TRANSIENT (in-memory only, rebuilt on demand, wiped by restart):
- `EncounterSession`, `GameState`, `heroBrains`, the combat phase machine, `DefendRound`, `MovementVote`, `aiPlayerBusy`/`phaseTransitioning`/`building`/`generation`, all timers (reap/afk/grace/vote/defend), `RoomCode`, `tokenIndex`, live `sessionToken` *bindings* (re-derived — see §6), `Seat.socket`, `Seat.ready/exhausted/actedThisPhase`.
- `room.phase==="combat"` is intentionally NOT durable. A restart mid-combat reconstructs the room at `overworld` with the party on `runs.party_q/party_r` (the tile they departed FROM — see §2.6 write ordering). The in-progress encounter simply does not exist; re-proposing the move re-enters it via the R7 atomic build.

### 2. Write points and transactionality (every durable write)

All durable writes are SQLite, on the single `bun:sqlite` `Database`. SQLite is single-writer; we wrap any multi-statement write in `db.transaction(...)`. Single-statement writes (one hex, one party-move) need no explicit transaction. Each write fires from the server **after** the in-memory state mutation succeeds, so durable state is a committed subset of in-memory state (never ahead of it). Writes are synchronous (bun:sqlite is synchronous) and small; no async gap.

Write points:

1. **Run create** (`createRoom` → first `startGame`, or on `createRoom` itself — see note): one transaction inserts `runs` row (`active=1`, `party_q/r=ORIGIN`, `host_client_id=creator`), seeds origin into `explored_hexes(run_id,...)` (replaces today's `seedDiscovery`/origin insert, now run-scoped), and inserts one `run_seats` row per occupied seat with its `client_id`. *Decision:* create the run row at **room create** time (so the host's clientId binds immediately and a restart during lobby still resumes), but only mark seats durable as humans actually take them (join) — see point 2.

2. **Seat bind / join / bot-fill** (a human joins, the host bot-fills on Start, or a bot seat is later reclaimed by a human): UPSERT the `run_seats` row (`client_id`, `display_name`, `controller_kind`), and snapshot that seat's *starter* inventory into `run_seat_items` in the same transaction (one transaction per seat). Bot-fill writes `controller_kind='bot'` with `client_id=NULL`.

3. **Equip / unequip / updateAttachment** (only legal while `room.phase!=="combat"`, R26): rewrite that seat's inventory rows. One transaction per mutation: `DELETE FROM run_seat_items WHERE run_id=? AND seat_index=?`, re-insert from the new in-memory `InventoryState` (bag entries by `id`+order, equipped entries by `id`+order), then upsert/delete the matching `run_seat_attachments`. Equip is the highest-frequency durable write; it is cheap (≤16 bag + a few equipped + a few attachment rows). Because equip is rejected during combat (R7/R26), there is never an inventory write concurrent with a combat build.

4. **Hex explore** (`exploreHex`, fired on `combatEnd{won:true}` with `pendingHex`): one transaction — `INSERT OR IGNORE INTO explored_hexes(run_id,q,r)` for the newly-explored hex (+ its icon row), AND `UPDATE runs SET party_q=?, party_r=?, updated_at=? WHERE id=?`. Party position and the explored set advance together atomically, so a restart can never show the party on an unexplored hex or an explored hex with stale position.

5. **Party move to a visited hex** (vote YES + already-explored target, no combat): single `UPDATE runs SET party_q=?, party_r=?, updated_at=?`. No transaction needed (one statement).

6. **Run complete / abandon / new run** (§5): `UPDATE runs SET active=0, completed_at=?, outcome=?` for the old run; INSERT a new `runs` row for the replacement. Done in one transaction so there is never a window with two active runs for the same party, nor zero. (See §5 for the matrix.)

NOT a write point: combat entry (R7 atomic build), defend, action, vote, ready/pass, host migration on disconnect, reconnect rebind. These are all transient. Host migration *does* lazily persist (point 7) so a restart re-derives a sane host.

7. **Host change** (migrateHost picks a new host, or the host leaves the lobby): `UPDATE runs SET host_client_id=? WHERE id=?` (single statement). This is a convenience write — on restart the host is re-derived anyway (§3), but persisting it keeps the "stays migrated" semantics (R14) stable across a restart that happens right after a migration.

Transactionality rule of thumb: any write that mutates >1 row or >1 table is a `db.transaction`. The party-position+explored-hex pairing (point 4) and the run-swap (point 6) are the two that *must* be atomic for correctness; the rest are atomic-by-being-single-statement.

### 3. Inventory durability representation

`InventoryState` holds full `ItemDefinition` objects, but those are fully reconstructable from `loadItems(dimensionId)` keyed by `id` (the existing `buildDefaultInventory` already does exactly this lookup). So we persist **IDs and ordering only**, never serialized item JSON:
- `run_seat_items(run_id, seat_index, location TEXT CHECK(location IN ('bag','equipped')), slot_order INTEGER, item_id TEXT)`. For `bag`, `slot_order` is the bag index 0..15 (empty bag slots simply have no row). For `equipped`, `slot_order` is the equipped-array index (ordering matters for `getAnimSet`/slot accounting).
- `run_seat_attachments(run_id, seat_index, item_id TEXT, attachment_json TEXT)` — the `AttachmentData` blob is small and not derivable, so it is stored as JSON keyed by item id (mirrors the in-memory `attachments: Record<itemId, AttachmentData>`).

Rehydration (`loadSeatInventory(runId, seatIndex, dimensionId) -> InventoryState`): `const merged = {...loadItems(0..3), ...loadItems(dimensionId)}` (same merge today), then build `bag` (length-16, nulls where no row), `equipped` (ordered), `attachments` map. **Item-not-found guard:** if a persisted `item_id` is missing from `merged` (item removed/renamed across a deploy), skip it and log; the seat loses that item rather than crashing the resume. This makes inventory resilient to content edits between restarts.

### 4. clientId <-> seat <-> run binding (the resume key)

The binding lives in `run_seats(run_id, seat_index, client_id, ...)`. The composite `(client_id)` is what a reconnecting socket presents (via its localStorage `clientId` in `hello`), and `(run_id, seat_index)` is what it resolves to. Lookups:
- `findActiveSeatForClient(clientId) -> { runId, seatIndex, dimensionId } | null`: `SELECT ... FROM run_seats JOIN runs ON runs.id=run_seats.run_id WHERE run_seats.client_id=? AND runs.active=1 AND run_seats.left_at IS NULL`. A client is in at most one active run/seat (enforced by application logic: on join we `left_at`-stamp any prior active seat for that client). Index: `CREATE INDEX idx_run_seats_client ON run_seats(client_id) WHERE left_at IS NULL`.

This is the table that makes restart-resume possible: after a restart, in-memory rooms/tokens are gone, but `hello{clientId}` finds the active run+seat and rebuilds.

### 5. Run lifecycle / inactivation matrix

`runs.active` is the single liveness flag (Phase-3 already adds it as part of `markRunInactive`). Transitions:
- **Victory that ends the run** (`gameOver outcome="victory"`): transaction sets old run `active=0, outcome='victory', completed_at=now`; does NOT auto-create a new run (party returns to lobby/menu; a fresh "play again" makes a new run via createRoom).
- **Party wipe / defeat** (`gameOver outcome="defeat"`, today's `resetToOrigin` "new run" path): transaction sets old run `active=0, outcome='defeat'`, INSERTs a new `runs` row (same dimension, same party = copy the human `run_seats` rows with fresh starter inventory, party at ORIGIN, `active=1`), and `room.runId` is repointed. This preserves today's "death = fresh run" behavior but durably.
- **Explicit reset by host** (`reset` in overworld): same as defeat path — old run `active=0, outcome='abandoned'`, new active run. In combat, `reset` aborts the encounter and returns to overworld of the *same* run (no run swap) — matches R17 reset semantics; nothing durable changes except party stays put.
- **Abandonment** (all humans leave the lobby/room, room disposed with no started run): the run row stays `active=1` so a player who comes back later still resumes. Stale active runs are not a correctness problem (resume just works); an optional housekeeping job can mark runs `active=0` after N days, but that is out of scope.
- **clientId joins a new room while having an active run**: on `createRoom`/`joinRoom` we first `left_at`-stamp that client's seat in any prior active run and, if that leaves the prior run with zero remaining human seats, mark it `active=0, outcome='abandoned'`. This keeps `findActiveSeatForClient` single-valued.

### 6. sessionToken with no in-memory state after restart (R5/R6)

The threat is seat-hijack: after a restart, the old in-memory token bindings are gone, so we must verify a reconnecting client owns its seat without trusting a self-asserted token. Two complementary mechanisms:

A. **Stable server secret + derivable token (preferred).** A `SERVER_SECRET` (env var, e.g. `GAME_TOKEN_SECRET`, persisted in the deployment env, NOT regenerated per boot) lets the server compute `sessionToken = HMAC(SERVER_SECRET, clientId + ":" + runId + ":s" + seatIndex)`. The token is deterministic and survives restart because both inputs are durable (`clientId`+`runId`+`seatIndex` from `run_seats`) and the secret is stable. On reclaim, the server recomputes the expected token for the `(clientId, runId, seatIndex)` it just looked up and compares constant-time against the presented token. No in-memory binding table is needed for *verification*. (We keep a transient `tokenIndex: Map<token, roomCode>` only as a fast path for the in-memory live-room case; on a miss it falls through to the DB lookup, R19's "tokenIndex hit whose room is missing → no room" still holds.)

B. **The clientId IS the resume credential; the token gates seat-hijack within a live room.** The reconnect identity check is: does `findActiveSeatForClient(presentedClientId)` resolve to a seat whose `client_id` equals the presented clientId? That binding is durable and authoritative. The HMAC token (mechanism A) is the *additional* anti-spoof layer so that a client cannot present someone else's `clientId` and steal their seat: it must also present a token that HMACs correctly, which only the legitimate owner received in its `welcome`. Because the token is HMAC-derived from the durable triple under a stable secret, a restart does not invalidate previously-issued tokens (no forced re-auth, no lost seats) — the central requirement of the product decision.

Resulting rules, restated for the post-restart case:
- A `hello`/`reclaimSeat` after restart presents `clientId` (+ token if the client cached one). Server: `findActiveSeatForClient(clientId)`; if found, recompute expected HMAC; if the client also presented a token it must match (else `SEAT_IN_USE`-style rejection / treat as fresh). The server returns the (re-derived, identical) token in `welcome` so the client's cached token and the server's expectation always converge.
- R6 single-owner still holds *within a reconstructed live room*: the reconstructed `Room` is the live registry of who has a socket. A second live connection for the same seat is `SEAT_IN_USE` unless `force`. Across a restart there are simply no live sockets yet, so the first reconnector wins the seat cleanly (see §7 race handling in resumeAlgorithm).
- Secret rotation = forced re-handshake for everyone (acceptable, rare); document that `GAME_TOKEN_SECRET` must be stable across restarts and is the one piece of config the durable-resume guarantee depends on. If absent, we fall back to a persisted `run_seats.session_token` column (random per seat, written at bind), which is mechanism A's data-driven equivalent — verification then compares the presented token to the stored column instead of recomputing. (Either is fine; HMAC avoids storing a secret per row.)

### 7. Reconstruction substrate (what gets read)

`reconstructRoomForRun(runId)` (server, in-memory, no combat):
- `SELECT * FROM runs WHERE id=? AND active=1` → dimensionId, capacity, party_q/r, host_client_id.
- `SELECT * FROM run_seats WHERE run_id=? ORDER BY seat_index` → seat membership, clientIds, controller_kind, displayName.
- `loadDiscoveredHexes(dimensionId)` + `loadDiscoveredHexIcons(dimensionId)` (GLOBAL) → `HexMapState{ playerPos:{q:party_q,r:party_r}, hexes, icons }`; `loadRunCleared(runId)` → `visitedThisRun` (per-run).
- per seat: `loadSeatInventory(runId, seatIndex, dimensionId)`.
- Construct a fresh `Room` with `phase="overworld"`, `building=false`, `generation=0`, `session=null`, `vote=null`, `defendRound=null`, all timers null, all seats `state` set per §3-resume (controller_kind → bot/disconnected), `socket=null`. Assign a **new** RoomCode from the registry (the old code is gone; resumers reach the run via clientId, not code, then can re-share the new code).

Combat is never reconstructed; `pendingHex` is null (a restart discards the in-progress encounter, and party_q/r already points at the safe departure tile).

### 8. Interaction with existing rulings

- **R7 atomic build (combat entry):** unchanged and *easier* under durability — combat entry remains the synchronous-flip-then-await build; we do not persist anything during the build. The durable party position was already written at the *previous* explore (point 4), and we do NOT advance party_q/r to the target until `combatEnd{won}` fires `exploreHex` (point 4). So a crash during the await leaves the durable run pointing at the departure tile, and resume + re-propose redoes the build. The `generation`/`building`/disposed re-check after the await is purely in-memory and untouched.
- **R13 run-scoped DB:** we *extend* it — every read/write is still `runId`-scoped; we add `run_seats`/`run_seat_items`/`run_seat_attachments` and the `runs` columns. The "no global-read caller remains" invariant is preserved; the new tables are all run-scoped too. The one-time `explored_hexes` wipe documented in Phase 3 still applies (the `user_version` bump now goes to a higher version that also creates the new tables).
- **R19 dispose:** `Room.dispose()` is unchanged for in-memory teardown (timers, tokenIndex, registry, quarantine). Dispose does NOT delete durable rows — the run stays `active=1` so it can be resumed. (Run inactivation is a separate, explicit lifecycle event, §5, not a side effect of dispose.) This decouples "room object gone" from "run over," which is exactly the restart-resume requirement.
- **R20 code quarantine:** unchanged and orthogonal — codes are in-memory and never the resume key. A reconstructed room gets a fresh code; the quarantined old code resolves to `ROOM_NOT_FOUND` (correct; resumers use clientId not code).
- **R21 seat-state machine:** on resume, `controller_kind='human'` seats start as `human-disconnected` (reclaim-only, never `open`); `controller_kind='bot'` seats start as `bot`. There are NO `open` seats in a reconstructed (already-started) run — matches R21's "post-Start there are no open seats; a new unknown-token join → ALREADY_STARTED." The synchronous check-then-flip on reclaim is unchanged (build inventory from DB into a local, commit synchronously). Lobby-only joinability is preserved because a resumed run is never in `lobby` phase.
- **R14 host model ("stays migrated"):** on resume the host is re-derived (§ resumeAlgorithm): lowest-index human seat that reconnects first becomes host; persisted `host_client_id` is a tiebreak/seed only, the original host does not bounce back if someone else reconnects first. `migrateHost()` continues to run on every human disconnect and lazily persists (write point 7).

## Resume / reconstruction algorithm

On hello/reconnect after a restart (in-memory rooms/tokens gone, durable run exists for this clientId):

1. hello{clientId, protocolVersion[, token]} arrives. Validate protocol; mismatch → protocolMismatch+close.
2. Fast path: check in-memory tokenIndex/registry. After a restart this is empty → fall through (R19: a tokenIndex hit whose room is missing is treated as no-room; here there is no hit at all).
3. DB path: row = findActiveSeatForClient(clientId). If null → welcome with no reconnect; client shows lobby (no active run).
4. row found → { runId, seatIndex, dimensionId, capacity }. Compute expectedToken = HMAC(SERVER_SECRET, clientId+":"+runId+":s"+seatIndex). If the client presented a token, constant-time compare; on mismatch treat as a fresh client for that seat only if no token was presented (legit first-reconnect after the client cleared storage), else reject SEAT_IN_USE-style. Always return the (re-derived, identical) token in welcome so client and server converge — no forced re-auth across restart.
5. Room reconstruction (idempotent, guarded): look up an in-memory Room already reconstructed for this runId (keyed by runId in a transient Map<runId, Room>). 
   a. If none exists: reconstructRoomForRun(runId) — read runs row, run_seats, loadExploredHexes(runId)+icons, per-seat loadSeatInventory. Build a fresh Room: phase='overworld', session=null, vote=null, defendRound=null, generation=0, building=false, all timers null, pendingHex=null, hexMap.playerPos={party_q,party_r}. Assign a NEW RoomCode (old code is gone). For each seat: controller_kind='human' → state='human-disconnected', socket=null, brain=null (reclaim-only, never 'open' per R21); controller_kind='bot' → state='bot', install sovereignFor(seat) brain. Register the Room (by code AND by runId).
   b. If one already exists (another client reconstructed it microseconds earlier): reuse it. Reconstruction must be synchronous from the point of the Map check to the Map insert (no await), so two near-simultaneous reconnections cannot both build — the first to run the sync block wins, the second sees the existing Room. (Bun's single-threaded event loop guarantees the check-build-insert runs to completion without interleaving as long as it contains no await; reconstruction reads are synchronous bun:sqlite calls, so this holds.)
6. Bind this socket to its seat: seat.socket=this; seat.state='human-connected'; seat.clientId=clientId; cancel any disconnectGraceTimer/reapTimer; if controller_kind was reconstructed as bot but this clientId owns it as human (controller_kind='human' in DB), drop the bot brain (R12 derivation re-enforces at next phase). Persist nothing here (rebind is transient).
7. Re-derive host (R14 'stays migrated'): if room.hostSeatId===null (true right after reconstruction), set hostSeatId = lowest-index seat whose state==='human-connected'. The FIRST human to reconnect becomes host regardless of persisted host_client_id (original host does not bounce back). Persisted host_client_id is only a seed/tiebreak if it matches a currently-connected seat at the instant of derivation; otherwise lowest-index-connected wins. Persist the new host (write point 7).
8. Send welcome{sessionToken, reconnected:{code,seatId}}, then roomState (roster shows other seats still human-disconnected until they reconnect), hexMapState (party on the durable tile), and inventory (this seat's bag) to this socket. Do NOT send combat state/coopStatus (phase is overworld; there is no session).

Two clients reconnecting near-simultaneously: both hit step 5; the sync check-build-insert in 5a/5b serializes them on the event loop, so exactly one Room is built and both bind into it (different seat_index rows → no seat contention). If two clients somehow present the SAME clientId (duplicate tabs), R6 applies post-reconstruction: the first binds the seat (state→human-connected, live socket); the second gets SEAT_IN_USE unless force (force closes the first socket, sends displaced). Host derivation runs after each bind, so whoever connected first to the lowest-index human seat is host; a later reconnection of a lower-index seat does NOT steal host (stays-migrated).

Mid-combat death → resume at overworld: if the server died while the party was inside an encounter, nothing combat-related was persisted. The durable run still points party_q/r at the DEPARTURE tile (party position is only advanced on combatEnd{won} via exploreHex, write point 4). So reconstruction places the party on the safe tile they came from; the unexplored target hex is still unexplored. The player re-proposes the move, and the R7 atomic build re-creates the encounter fresh. A hero that 'died' mid-combat simply never had that death persisted — the run state is exactly as it was before entering the encounter, which is the chosen boundary. If the entire party had wiped and the run-end had already committed (write point 6, active=0 + new active run inserted) BEFORE the crash, findActiveSeatForClient resolves to the NEW run (fresh party at ORIGIN), which is correct.

## Token verification after restart

After a restart there is no in-memory token binding table, so verification cannot rely on remembering which token was minted. Two stacked mechanisms make it work without trusting a self-asserted token (anti seat-hijack, R5/R6):

1. Stable server secret + derivable token (primary). A SERVER_SECRET env var (GAME_TOKEN_SECRET) is stable across restarts (NOT regenerated per boot). The sessionToken is HMAC(SERVER_SECRET, clientId + ':' + runId + ':s' + seatIndex). All three inputs are durable: clientId comes from hello, runId+seatIndex come from the durable run_seats row resolved by findActiveSeatForClient(clientId). On any reconnect the server RECOMPUTES the expected token from these durable inputs and constant-time-compares it against the token the client presents (the client caches the token it received in welcome). Because the secret is stable and the inputs are persisted, a token minted before the restart still verifies after it — no forced re-auth, no lost seats. The transient tokenIndex Map is kept only as a fast path for already-live rooms; on a miss it falls through to this DB-backed recompute (R19: tokenIndex hit with missing room → treat as no room).

2. Durable binding as the authority. The clientId<->seat binding is in run_seats and is authoritative for WHICH seat a reconnector may take. The HMAC token is the extra layer preventing a client from spoofing someone else's clientId: even with the right clientId string, an attacker cannot produce a valid token without the secret, so reclaim is rejected (SEAT_IN_USE / treated as unauthorized). R6 single-owner is enforced post-reconstruction by the live Room: the reconstructed Room is the registry of live sockets, so a second live connection for a seat that already has a socket is rejected unless force (which closes the old socket + sends displaced).

Fallback if a stable secret is undesirable: a per-seat random token persisted in a run_seats.session_token column, written at seat-bind; verification compares the presented token to the stored column. This is the data-driven equivalent of mechanism 1 and survives restart identically; the only cost is storing a secret per row. HMAC is preferred to avoid that. Either way, the single config dependency for the durable-resume guarantee is a stable secret/persisted-binding; secret rotation = a (rare, documented) forced re-handshake for everyone.

## Durable write points

1. Run create (createRoom): one transaction — INSERT runs(active=1, dimension_id, capacity, host_client_id=creator, party_q/r=ORIGIN, created_at/updated_at=now); seed origin into explored_hexes(run_id,...)+origin icon; INSERT run_seats row for the host seat (controller_kind='human', client_id=creator) + its starter inventory rows.
2. Seat bind / join / bot-fill (human joins, host bot-fills on Start, bot seat later reclaimed): one transaction per seat — UPSERT run_seats(client_id, display_name, controller_kind), and (on first human bind) snapshot starter inventory into run_seat_items. Bot-fill writes controller_kind='bot', client_id=NULL.
3. Equip (only legal while room.phase!=='combat', R26): one transaction — DELETE run_seat_items for (run_id,seat_index), re-INSERT bag+equipped rows from the new InventoryState by item id+order; upsert/delete matching run_seat_attachments rows. Highest-frequency durable write; cheap; never concurrent with a combat build because equip is rejected during combat (R7).
4. Unequip: same shape as equip — full rewrite of that seat's run_seat_items in one transaction; delete the removed item's run_seat_attachments row.
5. updateAttachment: one transaction — UPSERT run_seat_attachments(run_id,seat_index,item_id, attachment_json) to match the new in-memory attachments map.
6. Hex explore (exploreHex on combatEnd{won:true} with pendingHex): one transaction — INSERT OR IGNORE explored_hexes(run_id,q,r) + its icon row, AND UPDATE runs SET party_q=?, party_r=?, updated_at=?. Party position and explored set advance atomically (no window where party is on an unexplored hex).
7. Party move to already-visited hex (vote YES, no combat): single statement UPDATE runs SET party_q=?, party_r=?, updated_at=?. No transaction needed.
8. Run complete (gameOver victory): single statement UPDATE runs SET active=0, outcome='victory', completed_at=now. No new run auto-created (back to lobby/menu).
9. Run defeat / party wipe (today's resetToOrigin 'new run'): one transaction — UPDATE old run active=0, outcome='defeat'; INSERT new runs row (same dimension, party at ORIGIN, active=1); copy human run_seats with fresh starter inventory; repoint room.runId.
10. Host reset (host 'reset' in overworld): one transaction — UPDATE old run active=0, outcome='abandoned'; INSERT new active run as in the defeat path. (In-combat reset only aborts the encounter and returns to overworld of the same run — no run swap, no durable change.)
11. Host migration (migrateHost picks a new host, or host leaves lobby): single statement UPDATE runs SET host_client_id=? — convenience write so a restart right after a migration re-derives a sane host; host is re-derived from live sockets on resume regardless (R14 stays-migrated).
12. Client switches to a new room while holding an active run (createRoom/joinRoom): before binding, left_at-stamp this client's prior active run_seats row; if that leaves the prior run with zero human seats, UPDATE that run active=0, outcome='abandoned' — keeps findActiveSeatForClient single-valued.

## Stress-test findings (22; 9 high/critical) — resolved in the rulings above

1. **[high]** `visitedThisRun` (the set that decides whether entering a hex triggers combat) is NOT in the durable schema and reconstruction never rebuilds it. In the live code (index.ts:391) the combat-vs-free-move gate is `data.visitedThisRun.has(tk)`, and `visitedThisRun` is a separate set from `explored_hexes`: it is seeded as ONLY the origin (freshVisitedSet) while `explored_hexes` is seeded as a radius-15 disc of 'explored/visible' hexes (seedDiscovery(15)) and grown by exploreHex. The design's reconstructRoomForRun builds `HexMapState` purely from `loadExploredHexes(runId)` and party_q/r, and lists `visitedThisRun` as TRANSIENT/wiped-by-restart. So after a restart the room has no visitedThisRun, or if naively rebuilt from explored_hexes it would mark every visible-but-never-fought hex as 'already cleared'.
   - _why:_ This breaks the central correctness invariant of the resume. If visitedThisRun is rebuilt from explored_hexes, every previously-visible (radius-15 seeded) hex becomes a free walk with no encounter — the party walks the whole pre-seeded disc combat-free after any restart, trivially defeating the game loop. If visitedThisRun is reset to just origin, then hexes the party already cleared this run will re-trigger combat when re-entered, contradicting the design's own write-point-5 ('Party move to an already-explored target, no combat'). Either way the persisted state cannot reproduce the pre-crash combat gating, because the design persists only 'explored' (visibility) and not 'visited/cleared' (combat-done). The two sets are conflated throughout the delta (it repeatedly treats explored_hexes as both).
   - _fix:_ Persist the run-scoped cleared/visited set explicitly (e.g. a `cleared INTEGER` flag column on explored_hexes, or a separate run_cleared_hexes table), written in the same write-point-4 transaction as party position. Reconstruct visitedThisRun strictly from the cleared set, never from visibility. Document that 'explored' = visible and 'visited/cleared' = combat-resolved are distinct run-scoped sets, and that ONLY the cleared set gates re-combat. Seed origin as cleared at run create.

2. **[med]** Mid-combat death + the 'resume at overworld' rule still mis-places the party for a multi-hop move and for the move that ENTERED combat. The design says party_q/r is advanced 'only on combatEnd{won} via exploreHex' so a crash mid-combat leaves party on the departure tile. But proposeMove combat entry (R7, §6) sets `room.pendingHex=target` and flips phase synchronously; party_q/r is NOT updated. That is correct for the immediate departure tile. However the design never states that a YES vote on an ALREADY-VISITED target (write-point-5, single UPDATE party_q/r) is committed BEFORE any subsequent combat-triggering move. If a party votes through several visited hexes and then into an unexplored one in quick succession, only writes that actually fired are durable — fine — but the design's claim 'lands on the correct tile, not inside the encounter' assumes the move-to-visited UPDATE (point 5) and the combat-entry flip (no write) are correctly ordered. There is no write barrier guaranteeing the point-5 party UPDATE commits before combat entry mutates in-memory pendingHex; combat entry does no durable write, so if a crash happens after the in-memory party advanced to a visited hex but the point-5 single-statement UPDATE had not yet run, resume lands the party one hop back.
   - _why:_ The design's own rule 'each write fires AFTER the in-memory mutation succeeds, so durable is a committed subset of in-memory' is exactly the hazard: in-memory party position can be ahead of durable party_q/r for a visited-hex move, and combat entry reads in-memory position (the departure tile) which may be the un-persisted advanced tile. On crash, resume uses durable party_q/r (the stale one), then the player re-proposes — but they re-propose from a different tile than where combat was actually entered, so the unexplored target adjacency may differ. This is a narrow window but it violates the stated 'land on the correct tile' guarantee.
   - _fix:_ Make the visited-hex party move (point 5) and the combat-entry departure-tile commit explicit and ordered: before the R7 synchronous phase flip, ensure the current party_q/r is durably committed (the move-to-visited UPDATE must have run). Since bun:sqlite writes are synchronous, the simplest fix is to never let in-memory playerPos lead durable party_q/r — write party_q/r synchronously in the same handler turn as the in-memory move, before returning to the event loop where the next combat-entry message could be processed.

3. **[high]** The partial unique index `idx_run_seats_client ON run_seats(client_id) WHERE left_at IS NULL` will THROW a UNIQUE constraint violation on legitimate reconstruction/bind paths, crashing the resume. I verified in bun:sqlite that this index hard-enforces single-active-seat-per-client at the DB layer (a second INSERT with the same client_id and left_at NULL fails with 'UNIQUE constraint failed'). But §4 says single-valued-ness is 'enforced by application logic: on join we left_at-stamp any prior active seat.' The two mechanisms collide: the 'client switches to a new room' write-point left_at-stamps the prior seat THEN inserts the new one — if any code path inserts before stamping, or if a crash leaves a prior seat un-stamped (e.g. abandonment leaves the run active=1 with the seat left_at NULL forever per §5), the next createRoom/joinRoom for that client tries to UPSERT a new active run_seats row while the old one is still active=1/left_at NULL, and the unique index throws.
   - _why:_ §5 explicitly says abandoned runs stay active=1 and the seat's left_at is only stamped on an explicit join elsewhere. So a client that abandoned (closed tab) keeps a left_at=NULL row. When they later createRoom, the design left_at-stamps the prior seat first — but createRoom also INSERTs a brand-new runs row + run_seats row for the host seat in 'one transaction' (write-point-1). If the stamping of the OLD run is a separate transaction from the new-run creation (the matrix lists them as separate write points), there is a window, and more importantly the ordering of stamp-then-insert is not guaranteed across the two write points. A throw here is unhandled (bun:sqlite throws synchronously) and aborts the hello/createRoom handler, leaving the client unable to start a new game after any abandoned run.
   - _fix:_ Either drop the partial unique index and rely solely on application logic (the design body's stated approach), OR keep the index but make the left_at-stamp of the prior active seat part of the SAME transaction that inserts the new run_seats row, with the stamp ordered first, and wrap in a guard that tolerates the constraint. Add a test: client abandons run A (active=1, left_at NULL), then createRoom — assert no constraint violation and exactly one active seat.

4. **[med]** Attachments keyed by item_id collide when a seat carries two copies of the same item, and inventory rehydration cannot distinguish them. run_seat_attachments PK is (run_id, seat_index, item_id) and the in-memory model is `attachments: Record<itemId, AttachmentData>`. But run_seat_items stores multiple rows with the same item_id (e.g. two short-swords in bag slots 0 and 5, plus one equipped) distinguished by (location, slot_order). On rehydration the design rebuilds bag/equipped by id+order and the attachments map by id. If two equipped items share an id but have different attachment configs, the single attachment row overwrites; and if the same id is both equipped and in bag, the attachment applies ambiguously. STARTER_ITEM_IDS today are unique, but the design persists by id specifically to be content-edit-resilient and equip/unequip can produce duplicates over a run.
   - _why:_ The in-memory InventoryState itself keys attachments by item id (Record<string, AttachmentData>), so this ambiguity exists in-memory too — but the design's wholesale DELETE+reINSERT 'rewrite that seat's inventory rows ... by id+order' for run_seat_items vs single-row-per-id for attachments means the two tables have inconsistent cardinality. If the loadout ever contains duplicate item ids (resolver/inventory does not forbid it — equipFromBag just appends), the attachment association is lossy and a restart can equip an item with the wrong attachment, changing combat visuals/geometry. unequipItem deletes attachments[removedItem.id] which already drops the attachment for ALL copies of that id in-memory, confirming the id-keying is a pre-existing fragility the persistence layer faithfully reproduces but should not silently rely on.
   - _fix:_ Document and enforce that attachments are id-scoped by design (one config per item id per seat), and assert/validate at bind time that the loadout has no semantically-conflicting duplicate-id attachments; OR key attachments by the same (location, slot_order) tuple as run_seat_items so each physical item instance carries its own attachment. At minimum add a test with two same-id items to confirm rehydration is deterministic.

5. **[med]** Run-swap on defeat/reset (write-point-6/9/10) repoints room.runId in-memory, but a near-simultaneous restart between the DB transaction commit and the in-memory repoint leaves the reconstruction Map and findActiveSeatForClient pointing at the NEW run while a reconstructed Room (if any survived) or a racing reconnect could key the transient Map<runId, Room> by the OLD runId. The §7/resume algorithm keys the idempotency Map by runId; the run-swap changes runId. There is no statement that the in-memory Map<runId,Room> entry is re-keyed from old runId to new runId atomically with room.runId repoint. A reconnect that resolves findActiveSeatForClient to the NEW run while a stale Map entry under the OLD runId still holds the same Room object can build a SECOND Room for the new runId (split-brain across a run boundary), the exact failure the design claims to prevent only WITHIN a single runId.
   - _why:_ The split-brain mitigation (self-noted high risk) only reasons about two reconnects for the SAME runId racing reconstruction. It does not cover the run-swap case where runId itself changes under a live room. The transient Map<runId,Room> is the sole guard, and a defeat/reset rekeys runId; if the Map is not updated in the same synchronous block as room.runId, the guard is keyed on a stale id. Combined with the fact that defeat creates a NEW active run (so findActiveSeatForClient immediately resolves to it), a reconnect landing right at the swap can miss the existing Room.
   - _fix:_ On every run-swap (defeat/reset new-run), re-key the transient Map<runId,Room>: delete the old-runId entry and insert the new-runId entry in the same synchronous block that sets room.runId, with no await between. Add an assertion that registering a Room by runId fails if a different Room already exists for that runId, and that a run-swap moves the existing Room rather than orphaning it.

6. **[high]** Reconstruction with `host_client_id` re-derivation can leave a started run with NO controller and a deadlocked maybeEndPlayerPhase the moment combat is re-entered, if all reconstructed human seats are human-disconnected and the run had bot seats whose brains must drive the phase. §7 sets controller_kind='human' seats to state='human-disconnected' with brain=null on reconstruction, and host=null until a human reconnects. That is fine in overworld (no phase machine running). But the design's resume lands at overworld with the FIRST reconnector becoming host; if that human immediately proposes a move into an unexplored hex (R7 build) while OTHER human seats are still human-disconnected (brain=null, socket=null), startPlayerPhase runs with seats that are neither human-connected (no socket to act) nor bot (no brain). maybeEndPlayerPhase requires every seat ready||exhausted; a human-disconnected, brain-less seat is never driven and never readies. R9 only drives a seat human->bot AFTER a 3s grace on an actual disconnect event — but a reconstructed seat never fired a 'disconnect' event (it was born disconnected at restart), so no grace timer is armed and no human->bot flip is scheduled.
   - _why:_ R9/R28's human->bot flip is triggered by the close() handler arming a disconnectGraceTimer. A seat reconstructed as human-disconnected after a restart never went through close(), so nothing arms its grace timer or installs a Sovereign brain. If combat is entered before that human reconnects, the seat sits as human-disconnected with brain=null forever, and maybeEndPlayerPhase (R8) can never satisfy 'every seat ready||exhausted' — the player phase deadlocks. The design explicitly worries about deadlock in maybeEndPlayerPhase but only via the disconnect path, not the reconstructed-disconnected path.
   - _fix:_ On reconstruction, treat every human-disconnected seat as immediately subject to the grace/bot policy: either arm a disconnectGraceTimer at reconstruction so it flips to a Sovereign-driven bot after the grace window if no reconnect, or block combat entry (R7) until all seats are either human-connected or bot-driven. Simplest: at reconstruction, give human-disconnected seats a transient bot brain (state stays reclaim-only 'human-disconnected' but a brain drives them in combat) so maybeEndPlayerPhase can always complete; drop the brain on reclaim per R12.

7. **[low]** Equip/unequip wholesale DELETE+reINSERT of run_seat_items is NOT idempotent across a crash between the inventory write and an unrelated subsequent write, and the design's 'durable is a committed subset of in-memory' claim is violated for the equip case specifically because equip mutates in-memory state THEN writes — a crash after the in-memory equip but before the DELETE+INSERT transaction commits leaves the durable inventory one equip behind, and on resume the seat shows the OLD loadout while the player believed they equipped. For a SOLO/lobby flurry of equips this is benign, but if the equip that was lost is the one whose weapon the player then takes into combat, the reconstructed seat fights with a different loadout than the player last saw — and because combat snapshots loadouts into immutable SeatBuildSpec at R7 build time from in-memory (not durable) state, the live pre-crash combat used the new weapon while the resumed re-fight uses the stale one.
   - _why:_ This is the inherent in-memory-leads-durable window the design accepts ('writes fire after the in-memory mutation succeeds'). For party position it is mitigated by atomic pairing; for equip there is no mitigation and the self-noted risks only address throughput (thrash) and content-drift, not the lost-last-equip-on-crash case. The blast radius is small (lobby cadence, one equip) but it directly contradicts the stated invariant for the highest-frequency write, and the resumed combat re-fight can use a weaker/different loadout than the player chose.
   - _fix:_ Accept and document explicitly that the last in-flight equip can be lost on crash (the window is one synchronous handler turn, so in practice near-zero), OR send the client an inventory ack only after the durable write commits so the client UI never shows an equip the DB hasn't persisted. Since bun:sqlite is synchronous and the write fires in the same handler turn as the in-memory mutation, simply ordering the DB write before the sendInventory ack closes the visible-but-unpersisted gap.

8. **[med]** A completed run (outcome='victory', active=0) plus a reconnecting client whose ONLY run is that completed one resumes into NOTHING, but the design's findActiveSeatForClient(clientId) requires runs.active=1 — so the client correctly gets 'no active run / lobby'. The hazard is the OPPOSITE: §5 victory does NOT create a new run and returns 'to lobby/menu', yet the client's localStorage clientId is still bound (left_at IS NULL) to the now-inactive run_seats row. findActiveSeatForClient filters active=1 so it returns null — but the run_seats row for that client is never left_at-stamped on victory. The next createRoom left_at-stamps 'any prior ACTIVE run' (write-point-12), but this run is inactive, so the stamp is skipped and the dangling left_at=NULL row persists against an inactive run. This is harmless for resume but means the partial unique index (if kept) now permits the client to have a left_at=NULL row on an inactive run AND a left_at=NULL row on a new active run simultaneously — two rows, same client_id, both left_at NULL — violating the index.
   - _why:_ Verified the partial unique index is on client_id WHERE left_at IS NULL with NO active-run qualifier. After a victory, the seat row stays (active=0 run, left_at NULL). createRoom only stamps prior ACTIVE runs, so it does not stamp this inactive-run row, then inserts a new active-run seat row with the same client_id and left_at NULL. Two left_at-NULL rows for one client_id => UNIQUE constraint violation => createRoom throws => player cannot start a new game after winning. This is a concrete, reproducible crash on the common 'win then play again' path.
   - _fix:_ On run completion (victory/defeat/abandon), left_at-stamp ALL seat rows for that run in the same transaction that sets active=0, so no left_at-NULL rows survive against an inactive run. Then either the index or the application logic stays single-valued. Add a test: win a run, then createRoom, assert success and exactly one left_at-NULL row for the client.

9. **[med]** R20 reused/quarantined room codes after restart: a reconstructed Room is assigned a NEW code from the registry, but the in-memory quarantine map that prevents code reuse is ALSO transient (wiped by restart). So immediately after a restart, the registry's quarantine is empty and freshCode can hand out a code that, before the crash, was a live OR quarantined code for a DIFFERENT run. A player holding a stale invite code from before the restart will, post-restart, resolve that code to whatever NEW room happens to grab it from the empty-quarantine pool — the exact 'stale invite drops you into a stranger's room' hazard R20 was designed to prevent, reintroduced by the restart because quarantine state is not durable.
   - _why:_ R20's quarantine TTL relies on in-memory state that the design itself classifies as TRANSIENT ('RoomCode ... wiped by restart'). The design asserts 'quarantined old code resolves to ROOM_NOT_FOUND (correct; resumers use clientId not code)' — true for the OLD code immediately after restart (no room has it yet). But as new rooms reconstruct/create post-restart, freshCode draws from a 32^6 space with an EMPTY quarantine, so a recycled code can alias a pre-crash invite. The window is small (must collide in a 6-char space) but R20 explicitly exists because the product cares about this, and the design's claim that codes-are-orthogonal-to-resume does not cover the cross-restart code-aliasing case.
   - _fix:_ This is acceptable IF documented: after a restart, stale pre-crash codes are simply invalid until coincidentally recycled (probability ~room_count/32^6). State explicitly that invite codes do not survive restart and clients must re-share the new code (the design already says resumers use clientId, so the host re-shares). Optionally persist quarantine as (code, expires_at) rows so cross-restart reuse is still suppressed for the TTL. At minimum, add the assertion that a reconstructed Room never reuses a code currently held by another live in-memory Room.

10. **[low]** Two/three clients reconnecting within milliseconds: the design's synchronous check-build-insert serialization is correct ONLY if reconstructRoomForRun contains literally zero awaits, but loadSeatInventory per seat calls loadItems(0..3) + loadItems(dimensionId) which under the run-scoped DB (R13/Phase-3) and the asset path could be re-reading the items table 5x per seat x up-to-4 seats = up to 20 synchronous queries plus loadExploredHexes plus icon reads, all inside the critical section. While bun:sqlite is synchronous (verified), if ANY future refactor makes inventory/dimension loading async (e.g. lazy-loading item sprites or a remote items source), the no-await guarantee silently breaks and split-brain returns. The design hard-depends on an implementation detail (sync sqlite) for a concurrency-correctness invariant, with no runtime assertion that the critical section actually ran without yielding.
   - _why:_ The self-noted high risk acknowledges the split-brain and mitigates via 'no await; bun:sqlite reads are synchronous.' This is correct today (I verified bun:sqlite query/all/get are synchronous). But the invariant is load-bearing and fragile: it is enforced by convention, not by structure. The mitigation even says to 'add an assertion that registration by runId is rejected if a Room for that runId already exists' — that assertion is the real guard and should be the PRIMARY mechanism, not the secondary one, because it survives a future async refactor whereas 'we promise not to await' does not.
   - _fix:_ Make the Map<runId,Room> insert use a check-or-throw register (registerRoomForRun throws/returns-existing if runId already present) as the authoritative anti-split-brain guard, so even if a future await sneaks into reconstruction the second builder's register call loses cleanly. Keep the no-await property as defense-in-depth, and add a dev assertion (or test) that reconstructRoomForRun performs no await.

11. **[low]** Host re-derivation across restart can hand host to a seat whose clientId never reconnects, stranding host-gated actions, because §7 step 7 sets hostSeatId to the lowest-index seat whose state==='human-connected' AND falls back to persisted host_client_id only 'if it matches a currently-connected seat'. If the FIRST reconnector is a non-host seat and the persisted host never returns, host correctly migrates (good). But the design also says 'reconstructed seats start as human-disconnected; there are NO open seats.' If a 3-human run restarts and only ONE human (a non-host) reconnects, that human becomes host — correct. The actual gap: between reconstruction (hostSeatId=null) and the first bind, ANY host-gated message (startGame is impossible since not in lobby, but RESET and DEBUGWIN are host-gated and valid in overworld) sent by a client that reconnected but lost the host-derivation race will get NOT_HOST, AND if the reconstruction happened but no human has yet bound (e.g. all three are mid-handshake), hostSeatId=null and reset is impossible — a reconstructed overworld room with zero bound humans cannot be reset, only reaped.
   - _why:_ This is mostly correct-by-design (R14 stays-migrated, idle reaper handles host-less rooms), but the durable layer adds a new transient state — a reconstructed room with hostSeatId=null and all seats human-disconnected — that exists for a real (if brief) window where the room is fully un-actionable except via reap. The design covers host migration on disconnect but does not state how long a reconstructed room can sit host-less, nor that the reap timer is armed at reconstruction for a room nobody has reconnected to.
   - _fix:_ Arm the idle reap timer at reconstruction time (not only on the last human leaving), so a reconstructed room that nobody reconnects to is eventually reaped rather than lingering active=1 forever holding a code. Confirm that the first human bind both derives host and cancels the reap timer in the same synchronous block.

12. **[critical]** The HMAC token has NO secret per-seat salt/nonce, so it is fully forgeable by anyone who learns (or guesses) the relationship and ever obtains the secret — and worse, it is a pure function of NON-SECRET, ENUMERABLE inputs. Delta defines sessionToken = HMAC(SERVER_SECRET, clientId + ':' + runId + ':s' + seatIndex). The original R5 (DESIGN.md line 341) was HMAC(serverSecret, clientId + nonce) — a random per-binding nonce that is NOT derivable from public data. The delta DELETES the nonce. Now the ONLY secret protecting every seat in every run is the single global SERVER_SECRET. runId is a small AUTOINCREMENT integer (db.ts line 32: `id INTEGER PRIMARY KEY AUTOINCREMENT`), seatIndex is 0..3, and clientId is presented by the attacker. So if SERVER_SECRET ever leaks (env dump, log, crash report, source/CI exposure, a single compromised deploy), an attacker can mint a valid token for ANY (clientId, runId, seat) tuple and hijack every seat and its durable inventory in the entire database — there is no per-seat entropy to stop lateral movement. With the original nonce design, leaking the secret still left each seat protected by its own unguessable nonce.
   - _why:_ This converts a per-seat compromise into a total, permanent, retroactive compromise of all identities. Because the inputs are durable and enumerable, the blast radius of a secret leak is the entire historical run/seat space (and persisted inventories), not one session. The delta explicitly traded the nonce away to get determinism across restart, but determinism does not require dropping per-seat entropy.
   - _fix:_ Keep a per-seat secret nonce that is DURABLE but NOT derivable from public inputs: store run_seats.token_salt (random 16+ bytes, written at bind) and compute sessionToken = HMAC(SERVER_SECRET, clientId + ':' + runId + ':s' + seatIndex + ':' + token_salt). This survives restart (salt is persisted) AND keeps each seat individually unguessable so a secret leak does not become trivially enumerable forgery. This also collapses to the doc's own 'fallback' (random per-seat session_token column) — just make that the PRIMARY, not the fallback. The pure-derivable HMAC-without-salt should be removed entirely.

13. **[critical]** findActiveSeatForClient(clientId) is treated as authoritative for WHICH seat a reconnector takes (Resume step 3, §4), but clientId is a self-asserted localStorage value sent in plaintext in `hello{clientId}` (DESIGN.md line 201). The design's anti-spoof story is 'the HMAC token is the extra layer,' yet Resume step 4 contains a fatal escape hatch: 'if the client presented a token, constant-time compare; on mismatch treat as a fresh client for that seat only if NO token was presented (legit first-reconnect after the client cleared storage).' This means: present a victim's clientId WITH NO TOKEN and you are NOT rejected — you are 'treated as a fresh client for that seat.' The whole point of the token gate is defeated for the no-token case, which is exactly the attacker's case (the attacker never had the token).
   - _why:_ An attacker who learns a victim's clientId (it is in the victim's localStorage, may appear in client logs/diagnostics — line 101 literally calls clientId a diagnostic field — and is not treated as a secret anywhere) sends hello{clientId: victim, /* no token */} after a restart and is admitted to the victim's seat and durable inventory. clientId was never designed to be a secret (R5 line 25 calls it 'a random UUID, localStorage' and uses the TOKEN as the secret), so the no-token branch hands seats to anyone who knows a UUID.
   - _fix:_ Remove the no-token admission branch entirely for already-bound (post-Start / active-run) seats. A reconstructed run seat has controller_kind='human' with a real prior owner; reclaiming it MUST require a valid token, period. If the legitimate owner truly cleared storage and lost the token, that is an explicit account-recovery / forfeit path (seat stays human-disconnected, eventually bot-flips, or host re-invites), NOT a silent first-come admission. Concretely: in resume step 4, `if (row.controller_kind==='human' && !validToken) reject` with no 'treat as fresh' fallback.

14. **[high]** The mid-combat-crash boundary combined with R28's 3s grace/bot-flip creates a post-restart seat-theft + griefing window, and 'reclaim vs grace' is ill-defined after reconstruction. On a normal disconnect the seat shows human-disconnected for 3s then flips to bot. But after a RESTART, reconstruction (§7 / resume 5a) sets ALL human seats to 'human-disconnected' with NO grace timer armed and NO reap timer — the design says 'all timers null.' So a reconstructed human seat is reclaim-only INDEFINITELY with no bot-flip, which contradicts R28's liveness guarantee: if the rightful owner never returns, the seat is stuck human-disconnected forever and the party cannot progress (no bot drives it, no flip). Conversely, if any grace/flip IS armed at reconstruction, a reconstructed seat could bot-flip in 3s and then a returning legitimate owner races the bot — but the design is silent on which.
   - _why:_ R28 (line 75/349) makes the 3s-grace→bot-flip the mechanism that prevents one absent human from deadlocking maybeEndPlayerPhase / overworld votes. Reconstruction explicitly nulls all timers (§7, resume 5a) and never re-arms grace for the reconstructed human-disconnected seats, so the liveness invariant R28 guarantees in-process does not hold across restart. The boundary is genuinely undefined: 'all timers null' + 'state=human-disconnected' = a seat that is neither live nor reapable.
   - _fix:_ Define reconstruction liveness explicitly: a reconstructed run with at least one connected human should arm the SAME R28 grace/AFK machinery for its still-disconnected human seats (e.g., on first reconnect, arm a grace/flip timer for the other human-disconnected seats, or treat reconstructed-but-unclaimed human seats as bot-driven from t0 and let R10 reclaim-clears-ready logic hand control back when/if the human returns). Also arm a reap timer for a reconstructed room with ZERO connected humans so abandoned reconstructions are torn down rather than living forever. State the exact grace value and the reclaim-vs-flip race resolution (reclaim with valid token always wins over a pending flip; flip never invalidates the durable token).

15. **[high]** A single clientId can legitimately appear in TWO active runs across the create/join race, breaking the 'single-valued findActiveSeatForClient' invariant the resume algorithm depends on. The 'single active run per client' property is enforced by APPLICATION logic only (§4: 'on join we left_at-stamp any prior active seat'), not by a DB constraint. The partial index `idx_run_seats_client ON run_seats(client_id) WHERE left_at IS NULL` is NON-UNIQUE (TABLES entry says 'Partial index ... for findActiveSeatForClient', not a UNIQUE index). If the left_at-stamp of the prior seat and the insert of the new seat are not in ONE transaction (write point 12 describes them as sequential: 'before binding, left_at-stamp ... if that leaves the prior run with zero human seats, UPDATE active=0'), a crash or interleaving between the two writes — or two concurrent createRoom/joinRoom from duplicate tabs of the same clientId — leaves the client with two rows where left_at IS NULL across two active runs. findActiveSeatForClient then returns a nondeterministic/arbitrary row (the SELECT has no ORDER BY / LIMIT 1 tiebreak specified), so the same clientId resolves to a different seat depending on row order, and on restart the player may resume into the WRONG run/inventory.
   - _why:_ The entire resume substrate assumes findActiveSeatForClient is single-valued; the design even says so ('A client is in at most one active run/seat (enforced by application logic)'). Enforcing a uniqueness invariant in application logic across multiple non-atomic writes, with no DB-level guard and no deterministic SELECT, is exactly where the binding integrity breaks under restart/concurrency — the two failure modes the task asks about ('a clientId that appears in two active runs', 'two seats claiming one clientId').
   - _fix:_ Make it a real constraint: `CREATE UNIQUE INDEX idx_run_seats_client_live ON run_seats(client_id) WHERE client_id IS NOT NULL AND left_at IS NULL`. Put the prior-seat left_at-stamp + run-inactivation + new-seat insert in ONE db.transaction so there is never a window with two live rows. Give findActiveSeatForClient a deterministic tiebreak (ORDER BY runs.updated_at DESC, run_id DESC LIMIT 1) so even a residual duplicate resolves predictably to the newest run. Note client_id IS NULL (bot seats) must be excluded from the unique index or every bot row collides.

16. **[med]** R6 'single live socket' is reconstructed correctly for the FIRST reconnector but the split-brain guard relies on an unstated assumption that ALL identity entry points funnel through the same transient Map<runId, Room>. The design adds a second registry key ('Register the Room by code AND by runId', resume 5a) but the create/join path (createRoom/joinRoom by CODE) does NOT consult the runId map — and after a restart the old code is gone, so a returning player MUST come via clientId/hello. Problem: nothing prevents a player from creating a BRAND NEW room (new run) via createRoom while ALSO having an active prior run that another tab is resuming via hello. The createRoom path (write point 1) inserts a new runs row and a new run_seats row for the same clientId; write point 12 is supposed to left_at-stamp the prior seat, but if the prior run was ALREADY reconstructed in-memory (live Room keyed by runId) the DB left_at-stamp does NOT close the live socket bound to that reconstructed seat. So you get the same clientId live in two in-memory Rooms simultaneously (one reconstructed-by-runId, one created-by-code), violating R6's single-live-socket-per-identity intent across the two registries.
   - _why:_ R6 is enforced per-Room (a seat has one socket), but identity (clientId) can now span two Room objects because reconstruction and creation use different registry keys and the abandonment of the prior run is a DB-only write that does not tear down the prior run's live reconstructed Room. The single-live-socket invariant is per-seat, not per-identity, so the design's claim that R6 'still holds' after reconstruction is true only within one Room and misses the cross-Room identity case the restart introduces.
   - _fix:_ When left_at-stamping a client's prior active seat (write point 12), ALSO look up any in-memory reconstructed Room for that prior runId (the transient Map<runId,Room>) and, if the seat has a live socket, close it / send displaced and unbind — i.e., make the DB abandonment and the in-memory teardown atomic at the application level. Equivalently, route createRoom/joinRoom for an identity with an active run through the same reclaim/abandon path that resume uses, so there is exactly one place that owns the clientId->live-socket mapping.

17. **[high]** Durable per-seat inventory can be read/written by the wrong seat because all inventory mutation authorization is implicit in 'the seat that sent the message,' and that authorization is never re-validated against the durable binding at the DB write. Write points 3/4/5 (equip/unequip/updateAttachment) do `DELETE FROM run_seat_items WHERE run_id=? AND seat_index=?` then re-insert. The seat_index used is whatever the in-memory Seat the socket is bound to — but after a reconstruction-then-takeover sequence (force reclaim, or the no-token admission bug above), a socket can be bound to a seat it does not legitimately own, and the equip write will then overwrite the LEGITIMATE owner's durable inventory. There is no check at the write that the socket's authenticated clientId still equals run_seats.client_id for that (run_id, seat_index). The blast radius is durable: a wrongful equip permanently rewrites another player's persisted bag, surviving restart.
   - _why:_ Inventory is now durable and the write key is (run_id, seat_index), decoupled from the identity that proved ownership at hello. Any path that mis-binds a socket to a seat (force takeover sequencing, the no-token branch, a duplicate-clientId race) becomes a durable cross-seat inventory write, not just a transient mislabel. The design asserts equip is 'never concurrent with a combat build' but never asserts equip is 'only by the seat's authenticated owner at the moment of write.'
   - _fix:_ Gate every durable inventory write on a re-checked authorization: assert socket.data.clientId === (SELECT client_id FROM run_seats WHERE run_id=? AND seat_index=? AND left_at IS NULL) inside the same transaction, and reject otherwise. Bind inventory rows to the identity, or at minimum verify identity at write time. On force-takeover, ensure the displaced socket can no longer issue equips for that seat (clear its socket.data.seatId before sending displaced).

18. **[high]** Stable-server-secret handling has no rotation story beyond 'forced re-handshake for everyone,' but with the no-salt token (finding 1) a secret ROTATION silently RE-ENABLES old hijack tokens to fail-open if any code path still accepts the no-token branch, and a secret that is WEAK/short makes offline brute force feasible because the plaintext is fully known. Since token = HMAC(secret, clientId:runId:sN) with all of clientId/runId/seat known or enumerable, an attacker who captures ONE legitimate (clientId, runId, seat, token) tuple (e.g., from a victim's client logs / a shared-machine localStorage / a MITM of the first hello on a non-TLS dev/staging deploy) can mount an offline dictionary/brute-force attack on SERVER_SECRET — every captured tuple is a verifiable HMAC oracle. The design names this as a 'self-noted risk' only in the 'secret unset/regenerated' direction (availability), never the 'secret weak/recoverable' direction (confidentiality of the master key).
   - _why:_ A keyed MAC whose entire message is attacker-known degrades to 'is the key strong enough to resist offline attack,' and the design specifies no minimum entropy, no KDF, and no per-seat salt to blunt it. One leaked tuple threatens the master secret for the whole system. This is a direct consequence of dropping the nonce and is distinct from the availability risk the doc already lists.
   - _fix:_ Require SERVER_SECRET to be high-entropy (>= 256 bits, generated, not a passphrase) and fail fast if shorter; combine with the per-seat salt from finding 1 so a captured tuple is not a clean oracle for the global key. Mandate TLS for the hello handshake (token only travels in first hello per R5, but only TLS makes that meaningful). Document rotation as: bump a secret-version, keep accepting old-version tokens for a grace window keyed by a persisted version tag, so rotation is not an all-or-nothing flag day that strands everyone.

19. **[med]** GDPR / privacy footgun: clientId (a stable per-user pseudonymous identifier) is now PERSISTED indefinitely in run_seats, joined to display_name, and the design explicitly says rows are NEVER deleted ('dispose does NOT delete durable rows', 'runs/run_seats tables grow unbounded', §8 R19). Combined with explored_hexes/party position/inventory, run_seats(client_id, display_name, joined_at, left_at) is a durable, long-lived behavioral profile keyed to a persistent device identifier with no retention limit, no deletion path, and no subject-erasure mechanism. The only cited cleanup is an 'optional, out-of-scope' housekeeping job keyed on completed_at — which never fires for the design's own 'abandoned run stays active=1 forever' case.
   - _why:_ A stable localStorage UUID tied to display name and timestamps is personal data under GDPR/CCPA; persisting it forever with no erasure path and explicitly unbounded growth is both a compliance liability and an operational one. The design treats unbounded growth purely as a storage concern and never as a data-protection concern, and the 'never resume key in URL' care taken for the token is undercut by storing the long-lived clientId in cleartext rows.
   - _fix:_ Add a retention/erasure path: a deletion routine that purges run_seats/run_seat_items/run_seat_attachments/runs for a given clientId (right-to-erasure) and a TTL housekeeping job that actually runs (not optional) to inactivate AND eventually delete completed/abandoned runs past N days. Consider salting/hashing clientId at rest if the raw UUID is only needed for equality lookup (store HMAC(secret, clientId) as the lookup key so a DB dump alone is not a clean device-tracking dataset) — though note this interacts with finding 1's token derivation and must be designed together.

20. **[med]** run_seats binding integrity has no guard against a clientId-with-no-seat or a controller_kind/client_id inconsistency, so reconstruction can produce a phantom human seat or silently drop a real one. The schema allows client_id NULL with controller_kind='human' (CHECK only constrains controller_kind to human/bot; nothing ties controller_kind='human' to client_id NOT NULL, nor controller_kind='bot' to client_id IS NULL). Write point 2 says bot-fill writes controller_kind='bot', client_id=NULL, and human bind writes client_id=creator — but nothing enforces it. A row with controller_kind='human', client_id=NULL (e.g., a partial/crashed bind transaction, or a human seat whose client_id was left_at-stamped to NULL by a buggy abandon) reconstructs (resume 5a: 'controller_kind=human → state=human-disconnected, reclaim-only') into a seat that NO clientId can ever reclaim (findActiveSeatForClient matches on client_id=?, NULL matches nobody) and that never bot-flips (finding 3) — a permanently dead seat that may block party progress.
   - _why:_ The reconstruction decision (human-disconnected vs bot) keys off controller_kind, but reclaimability keys off client_id; the schema lets these two columns disagree. A human seat with NULL client_id is unreclaimable AND undrivable, which is a binding-integrity hole the task specifically calls out ('a clientId with no seat' / 'a seat ... with no clientId').
   - _fix:_ Enforce consistency at the schema level: CHECK((controller_kind='bot' AND client_id IS NULL) OR (controller_kind='human' AND client_id IS NOT NULL)) so a human row without a clientId cannot exist. In reconstruction, treat any human seat whose reclaim is impossible as bot-driven for liveness. Make seat binds and abandons fully transactional so a partial write cannot leave a half-bound human row.

21. **[med]** The win-then-crash run-swap race can resurrect a completed run or strand a player between the old and new run, because run completion/defeat (write point 9, §5 defeat path) is 'UPDATE old active=0 + INSERT new active=1 + copy human run_seats with fresh inventory + repoint room.runId' in one DB transaction, but room.runId repoint is IN-MEMORY and outside the DB transaction's durability. If the server crashes AFTER the DB transaction commits (old inactive, new active) but the client had already cached a token for the OLD run (token = HMAC(secret, clientId:OLD_runId:sN)), then on resume the client presents the OLD-run token while findActiveSeatForClient resolves to the NEW run (correct per §5). The recomputed expected token uses NEW runId, so the client's cached OLD-run token MISMATCHES — and per resume step 4, since a token WAS presented and it mismatches, the client is rejected SEAT_IN_USE-style and CANNOT resume its own fresh run. The legitimate owner is locked out of the new run by their own stale token.
   - _why:_ Because the token is bound to runId (finding 1's no-salt derivation makes runId part of the MAC message), a legitimate run-swap rotates every player's token, but the client only learns the new token from a `welcome` it never receives if it crashed before reconnecting. The 'mismatch with a presented token => reject' rule then turns a normal death-into-fresh-run into a self-lockout. The design claims 'a token minted before the restart still verifies after it' — true only if runId is unchanged; the defeat/reset paths deliberately change runId.
   - _fix:_ Do not reject on token mismatch when the mismatch is explained by a run-swap for the same clientId: on mismatch, re-derive the expected token for the resolved (new) run and, since the clientId binding is authoritative and the seat has no live socket, issue a fresh welcome with the new token (the client adopts it) rather than rejecting. I.e., token mismatch should only harden the LIVE-room same-run hijack case (R6), never block a clientId from its own authoritative seat after a legitimate run rotation. Equivalently, decouple the token from runId (bind to clientId+seat-salt only) so a run-swap does not rotate it.

22. **[low]** Constant-time compare and token verification are specified but the 'fast path tokenIndex then fall through to DB' logic (TOKEN AFTER RESTART mechanism 1, resume step 2) has a timing/oracle and a fail-open risk: a tokenIndex MISS 'falls through to DB-backed recompute,' but the recompute path computes expectedToken from findActiveSeatForClient(clientId) — if that returns NULL (no active run) the design (resume step 3) sends 'welcome with no reconnect.' There is no statement that a PRESENTED token for a clientId with no active run is rejected or even examined; it is silently ignored and the client is dropped to lobby. That is benign for the legit case but means the token is only ever checked when a row exists — so the security check is data-dependent and an attacker probing many clientIds learns (via reconnect vs lobby response and response timing) WHICH clientIds currently have active runs, an enumeration oracle over the user base.
   - _why:_ Distinguishable responses (reconnected vs lobby) and differing work (HMAC recompute only when a row exists) leak active-run membership for a probed clientId, enabling targeted hijack attempts (combine with finding 2's no-token branch). Minor on its own but compounds the higher-severity identity findings into a practical attack chain.
   - _fix:_ Make the no-active-run and the invalid-token responses indistinguishable in shape and timing: always perform a dummy constant-time HMAC comparison even when no row is found, and return the same generic welcome-to-lobby payload whether the clientId is unknown, has no active run, or presented a bad token. Avoid an observable 'this clientId has a run' signal to unauthenticated probes.
